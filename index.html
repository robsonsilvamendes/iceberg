A camada que deu superpoderes de banco de dados a uma pasta de arquivos Parquet

Apache Iceberg é um formato de tabela (table format) open-source, criado em 2018 na Netflix (Ryan Blue e Daniel Weeks) e hoje um projeto de topo da Apache Software Foundation. Ele não substitui o Parquet — ele organiza metadados por cima de arquivos Parquet, ORC ou Avro para dar a eles transações ACID, time travel, evolução de schema e de partição sem reescrever a tabela inteira. É o núcleo técnico por trás do conceito de Lakehouse, suportado por Spark, Trino, Flink, Snowflake, Databricks, BigQuery e praticamente todo motor analítico moderno. Nesta aula vamos abrir essa camada de metadados por dentro: snapshots, manifests, partition evolution e as garantias que tornam um data lake seguro como um banco de dados.

Antes do Iceberg
🧩 O problema: tabelas como "pastas com arquivos"

Durante anos, uma "tabela" em um data lake era, na prática, apenas uma convenção: uma pasta no S3/HDFS com arquivos Parquet dentro, organizada em subpastas ano=2024/mes=01/ (estilo Hive). O Hive Metastore guardava só o schema e o caminho raiz — para saber quais arquivos pertencem à tabela agora, o motor de consulta precisava listar o diretório toda vez. Essa convenção simples esconde uma pilha de problemas sérios em produção.

DesafioTabela Hive (diretórios)Apache Iceberg
Como saber quais arquivos pertencem à tabela?Lista o diretório (list em S3 — lento e sujeito a inconsistência)Lê uma lista explícita no manifest — zero listagem
Commit é atômico?Não — grava arquivos e atualiza partição em passos separadosSim — troca do ponteiro do metadata.json é atômica
Time travel / rollback?Não nativoSim — cada commit vira um snapshot navegável
Renomear uma coluna é seguro?Não — o match é por nome, rename pode confundir dadosSim — cada campo tem um ID estável (veja seção de Schema Evolution)
Trocar a estratégia de partição sem reescrever tudo?Não — muda a partição, reprocessa a tabela inteiraSim — partition evolution nativa
Pular arquivos irrelevantes sem abri-los?Só estatísticas internas do próprio Parquet (por arquivo)Estatísticas agregadas no manifest, por arquivo, sem abrir nenhum
A ideia central do Iceberg

Uma tabela deixa de ser "uma pasta que eu listo" e passa a ser uma lista explícita e versionada de arquivos, mantida em uma árvore de metadados desacoplada dos dados. Cada mudança na tabela — um INSERT, um DELETE, uma compactação — gera uma nova versão completa dessa lista, nunca edita a anterior. É essa imutabilidade que sustenta transações, time travel e evolução segura.

Fundamentos
🧊 Formato de tabela, não formato de arquivo

É comum confundir os dois. Parquet define como bytes de linhas e colunas ficam organizados dentro de um único arquivo — isso não muda com o Iceberg. O Iceberg define como um conjunto desses arquivos vira uma tabela consultável com SQL: como saber quais arquivos existem agora, quais existiam ontem, qual é o schema atual, e como pular arquivos que não interessam a uma query. Clique nas camadas abaixo para ver onde cada peça do stack entra em ação.

👆 Clique em qualquer camada
O Iceberg vive entre o motor de consulta e os arquivos físicos — ele é a "cola" que dá semântica de tabela a um punhado de arquivos Parquet espalhados no armazenamento.
📄 Formato de Arquivo (Parquet, ORC, Avro)
Resolve "como comprimir e organizar bytes dentro de um arquivo". Row groups, encodings, compressão — tudo o que vimos na aula de Parquet continua valendo, inalterado, por baixo de uma tabela Iceberg.
🧊 Formato de Tabela (Iceberg)
Resolve "como um conjunto de arquivos vira uma tabela". Não guarda uma única linha de dado — só metadados: schema, partições, snapshots e a lista exata de arquivos válidos em cada momento.

👆 Clique em qualquer bloco
Um leitor sempre começa pelo catálogo — nunca lista o diretório de dados. Cada seta neste diagrama é um ponteiro explícito guardado em algum arquivo de metadados.

💡 Repare que a leitura é sempre em cascata, de cima para baixo: (1) pergunta ao catálogo qual é o metadata.json atual; (2) lê o current-snapshot-id dentro dele e encontra o manifest list correspondente; (3) lê o manifest list e decide, por estatísticas de partição, quais manifests abrir; (4) dentro de cada manifest, decide, por estatísticas de coluna, quais arquivos de dados abrir; (5) só então abre bytes de Parquet de verdade. Nenhum passo exige listar uma pasta.

📋 metadata.json
O arquivo raiz. Guarda table-uuid, location, todo o histórico de schemas e partition-specs (com IDs), current-snapshot-id, a lista completa de snapshots[] e o snapshot-log (para time travel por timestamp). É reescrito por completo a cada commit — nunca editado in-place.
📜 Manifest List (Avro)
Um arquivo por snapshot. Lista os manifests que compõem aquele snapshot e, para cada um, guarda um resumo por campo de partição (lower_bound/upper_bound) — é a primeira camada de pruning, antes de abrir qualquer manifest.
📄 Manifest File (Avro)
Lista arquivos de dados (ou delete files) escritos juntos. Para cada arquivo: caminho, valor de partição, contagem de linhas, tamanho e lower/upper bound por coluna — a segunda camada de pruning, no nível de arquivo individual.
Histórico Imutável
⏱️ Snapshots e Time Travel

Cada operação de escrita — INSERT, DELETE, MERGE, até uma compactação de manutenção — gera um snapshot novo: um novo manifest list + um novo metadata.json apontando para ele. Snapshots antigos continuam válidos e legíveis até serem explicitamente expirados. Clique em um ponto da linha do tempo abaixo para "viajar no tempo" até aquele commit.

Adeus, Coluna de Partição Manual
🙈 Hidden Partitioning e Transforms

Em uma tabela Hive, particionar por dia exige criar e manter manualmente uma coluna dt derivada de evento_ts — e o usuário precisa lembrar de filtrar por dt, não só por evento_ts, ou o pruning simplesmente não acontece. O Iceberg elimina essa coluna extra: a transformação de partição fica registrada no partition spec, e o motor de consulta traduz sozinho um filtro em evento_ts para pruning de partição. Isso é hidden partitioning.

📅 year / month / day / hour
Deriva a partição diretamente de uma coluna de data/timestamp. Ideal para dados com padrão temporal de chegada e de consulta (a maioria dos pipelines).
🪣 bucket(N, col)
Aplica hash e distribui em N baldes fixos. Ótimo para colunas de alta cardinalidade sem ordem natural (ex: user_id), evitando partições desbalanceadas.
✂️ truncate(N, col)
Trunca uma string ou número para os primeiros N caracteres/dígitos — útil para agrupar por prefixo (ex: os 2 primeiros dígitos de um CEP).
O usuário nunca precisa saber disso

Uma query WHERE evento_ts >= '2026-07-01' funciona sem que ninguém escreva WHERE dt = '2026-07'. E, como veremos a seguir, você pode trocar de month(evento_ts) para day(evento_ts) no futuro sem reescrever um único arquivo antigo.

Segurança de Longo Prazo
🧬 Evolução de Schema por Field ID

Cada coluna (inclusive campos aninhados) recebe, na criação, um ID inteiro permanente — gravado junto com os dados físicos no arquivo Parquet. O schema "atual" da tabela é só um mapeamento de nomes para esses IDs. Renomear, adicionar, remover ou reordenar colunas altera apenas esse mapeamento no metadata.json: os arquivos antigos nunca são reescritos e continuam sendo lidos corretamente, porque o leitor casa colunas por ID, nunca por nome ou posição.

Sem Reescrever a Tabela
🗺️ Evolução de Partição

Numa tabela Hive, mudar o esquema de particionamento significa reprocessar e reescrever toda a tabela. No Iceberg, um ALTER TABLE ... REPLACE PARTITION FIELD só afeta os dados escritos daqui pra frente — cada manifest guarda um partition_spec_id, e o planejador de query mistura specs diferentes de forma transparente na mesma tabela.

Como um DELETE/UPDATE Acontece
✍️ Copy-on-Write vs. Merge-on-Read

Como o Iceberg nunca edita um arquivo Parquet in-place, um DELETE/UPDATE/MERGE precisa escolher uma estratégia. Experimente rodar DELETE FROM vendas WHERE pais = 'Portugal' nos dois modos e compare o custo de escrita.

Estratégia
Operação simulada
DELETE FROM vendas WHERE pais = 'Portugal'
Bytes escritos agora—
Custo extra em leituras futuras—
📝 Copy-on-Write (COW)
Reescreve por completo cada arquivo afetado, sem as linhas removidas/atualizadas. Escrita mais cara, mas as leituras seguintes não têm nenhum custo extra — os dados já estão "limpos".
🔀 Merge-on-Read (MOR)
Grava só um pequeno delete file (position delete ou equality delete) apontando para as linhas removidas. Escrita muito mais barata, mas toda leitura futura precisa mesclar dado + delete file — até a compactação absorver essa dívida.
Sem Locks Longos
🔒 ACID via Concorrência Otimista

O Iceberg não trava a tabela enquanto um job escreve dados — isso poderia levar minutos ou horas. Em vez disso, usa concorrência otimista: o escritor prepara tudo em paralelo e só no final tenta uma troca atômica (compare-and-swap) do ponteiro no catálogo.

pseudo-código# 1. lê a versão base sem travar nada base = catalog.current_metadata_pointer("db.vendas") # 2. escreve novos data files + manifests + manifest list + metadata.json # (pode levar minutos — ninguém mais é bloqueado durante isso) new_metadata = build_new_metadata(base, changes) # 3. tenta a troca atômica do ponteiro (compare-and-swap) ok = catalog.compare_and_swap("db.vendas", base, new_metadata) if not ok: # alguém commitou primeiro — refaz sobre o snapshot mais novo retry()
🏁 Cenário: dois writers concorrentes
Writer A e Writer B partem ambos do snapshot v5. A termina primeiro e faz o CAS com sucesso → tabela vai para v6. Quando B tenta o CAS contra v5, ele falha (a base já não é a atual) — B recalcula seus manifests contra v6 e tenta de novo, chegando a v7.
⚡ Por que isso escala bem
O trabalho pesado — escrever arquivos Parquet e manifests — acontece sem nenhum lock. Só a etapa final (trocar um ponteiro de alguns bytes) precisa ser atômica, e isso o catálogo resolve em milissegundos.

⚠️ Isso significa que o catálogo escolhido precisa suportar compare-and-swap atômico de verdade — é por isso que "um arquivo texto qualquer apontando pro metadata.json" não serve como catálogo em produção multi-writer.

A Tabela Precisa de Cuidado
🧹 Manutenção: compaction, expiração e limpeza

Uma tabela Iceberg viva acumula arquivos pequenos, delete files e snapshots antigos. Sem manutenção periódica, o desempenho de leitura degrada e o custo de armazenamento sobe.

Experimente: compactação de arquivos pequenos (rewrite_data_files)

Tamanho médio atual6 MB
🧹 rewrite_data_files
Compacta arquivos pequenos (e absorve delete files do MOR) em arquivos maiores e mais eficientes de ler. A prática mais impactante de manutenção.
🗑️ expire_snapshots
Remove snapshots mais velhos que uma janela de retenção, junto com os arquivos que só existiam para servi-los — é o que finalmente libera espaço em disco após um rollback ou compactação.
🧽 remove_orphan_files
Limpa arquivos presentes no armazenamento mas não referenciados por nenhum manifest vivo — sobras de jobs que falharam no meio da escrita.
A Única Fonte da Verdade
📚 Catálogos

O catálogo tem uma única responsabilidade crítica: mapear nome_da_tabela → localização do metadata.json atual, com suporte a compare-and-swap atômico. Sem essa garantia, dois escritores concorrentes podem corromper a tabela.

🐘 Hive Metastore
O catálogo clássico do ecossistema Hadoop. Amplamente suportado, mas thrift-based e operacionalmente mais pesado de manter.
☁️ AWS Glue
Catálogo gerenciado da AWS, integrado nativamente com Athena, EMR e Redshift Spectrum.
🌐 REST Catalog
O padrão moderno: uma API HTTP especificada pelo próprio projeto Iceberg. Implementada por Tabular, Polaris, Unity Catalog, Snowflake e outros — o caminho para portabilidade real entre motores.
🌳 Nessie
Catálogo com semântica de Git — branches e tags cobrindo múltiplas tabelas ao mesmo tempo, permitindo transações multi-tabela e workflows de revisão de dados.

Mão na Massa
🐍 Iceberg na prática

De DDL em Spark SQL a leitura direta com PyIceberg — o essencial do dia a dia.

Spark SQL — criar com hidden partitioning
sqlcopiarCREATE TABLE db.vendas ( pedido_id BIGINT, cliente STRING, pais STRING, evento_ts TIMESTAMP, valor DOUBLE ) USING iceberg PARTITIONED BY (days(evento_ts)); ALTER TABLE db.vendas REPLACE PARTITION FIELD days(evento_ts) WITH hours(evento_ts);
Spark SQL — MERGE (upsert) e DELETE
sqlcopiarMERGE INTO db.vendas t USING stage_updates s ON t.pedido_id = s.pedido_id WHEN MATCHED THEN UPDATE SET * WHEN NOT MATCHED THEN INSERT *; DELETE FROM db.vendas WHERE pais = 'Portugal';
PyIceberg — inspecionando metadados
pythoncopiarfrom pyiceberg.catalog import load_catalog catalog = load_catalog("rest") tbl = catalog.load_table("db.vendas") print(tbl.schema()) print(tbl.current_snapshot().snapshot_id) df = tbl.scan(row_filter="pais == 'Brasil'").to_pandas()
Manutenção via Spark procedures
sqlcopiarCALL spark_catalog.system.rewrite_data_files( table => 'db.vendas', options => map('target-file-size-bytes','536870912') ); CALL spark_catalog.system.expire_snapshots( table => 'db.vendas', older_than => TIMESTAMP '2026-06-01 00:00:00' );
Checklist
✅ Boas práticas
🪣 Escolha o transform certo
Prefira granularidade que gere partições de algumas centenas de MB a poucos GB. Hidden partitioning não elimina o problema de over-partitioning — só esconde a coluna.
🧹 Agende compactação
Rode rewrite_data_files regularmente, especialmente em tabelas com streaming/MOR — delete files não compactados degradam leitura progressivamente.
✍️ COW vs MOR conforme o padrão de carga
MOR para pipelines com muitos updates/deletes pequenos e frequentes (CDC); COW para tabelas lidas com muito mais frequência do que são alteradas.
🌐 Prefira um REST Catalog
Reduz o acoplamento a um único motor e é o padrão que o próprio projeto Iceberg está consolidando — mais portabilidade entre Spark, Trino, Flink e data warehouses.
